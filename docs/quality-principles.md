# 単語データ品質原則

> 最終更新: 2026-04-08（第3版 — frequencyTier実装に伴う語数方針改訂）

## 第一原則（最優先事項）

**「そのコースを勉強したら、その試験に合格できる / そのスコアを狙える」**

- 英検○級コースを学習した人は、英検○級に合格できる語彙力を身につける
- TOEIC○○○コースを学習した人は、TOEIC○○○点を狙える語彙力を身につける
- 中学○年コースを学習した人は、中学○年レベルの定期テスト・高校受験に対応できる
- 高校○年コースを学習した人は、高校○年レベルの定期テスト・大学受験に対応できる
- 会話コースを学習した人は、そのレベルの英会話ができるようになる

**この原則に反するデータは、品質テストが通っていても不適切である。**

---

## 品質基準（優先順）

1. **meaningの正確さ（最優先）** — 学習者が間違った意味を覚えることは絶対に避ける。meaningは辞書的に正しく、試験で問われる意味と一致すること
2. **試験頻出度** — その試験・テストで実際に出題される語であること。語数の多さではなく、試験に出る語を正確に選ぶことが最優先
3. **レベル適合性** — そのステージの学習者にとって適切な難易度であること
4. **例文の自然さ** — ネイティブが自然に使う表現であること
5. **アルファベット分布** — 偏りの解消は上記1〜4を満たした上での追加目標（品質を犠牲にして偏りを埋めてはならない）

---

## meaning衝突の方針

**meaningの正確さが最優先。衝突解消のためにmeaningを歪めてはならない。**

同一ステージ内で同じmeaningの語が存在する場合:
- **コード側で対応する**: クイズの選択肢生成時に同一meaningの語を除外するロジックは実装済み（generator.ts, speed-challenge-logic.ts, review-quiz.ts）
- meaningを無理に変えて衝突を解消するのは禁止（cab=キャブのような不正確な訳が発生する原因になった）
- 同義語はmeaningが同じでも正しい。学習者が両方の語を覚えることに価値がある

---

## コース別のデータソース方針

**各コースは正しいソースから語を選定する。英検データの安易な転用は禁止。**

| コース | 正しいソース | 根拠 |
|--------|------------|------|
| 中学 | 学習指導要領 + CEFR-J Wordlist | 教科書準拠 |
| 高校 | 学習指導要領 + CEFR-J Wordlist | 教科書準拠 |
| 英検 | CEFR-J + 英検CEFR対照表 + 過去問分析 | 英検公式基準 |
| TOEIC | NGSL + TSL + BSL | TOEIC特化語彙リスト |
| 会話 | NGSL-Spoken + CEFR-J | 口語英語特化 |

### 禁止される転用パターン
- 英検データをそのまま高校コースに入れる（英検と高校は試験目的が異なる）
- 英検データをそのままTOEICコースに入れる（ビジネス語彙とのミスマッチ）
- 英検データをそのまま会話コースに入れる（学術例文と会話例文は異なる）

### 許可される転用
- コース間で同じ語が存在すること自体は許可（"go"は英検にも中学にもあってよい）
- 統合データモデルでは、1つのエントリが複数コースに所属する（courses配列）

---

## コース間の語の共有について（統合データモデル）

> **2026-04-08改訂**: データ統合（案A）により、コース間の重複問題を根本解決。

**1語1エントリの原則:**

- 同じ word + partOfSpeech は**1つのエントリ（1つのID）のみ存在する**
- 複数コースへの所属は `courses` 配列で表現する
- meaningは**全コースで統一**（コースごとに異なるmeaningは許可しない）
- 例文・カテゴリも1セット（コースごとに別の例文は持たない）
- 学習記録はID単位 → 1語1IDなので、コースをまたいでも**学習記録が自動的に統合される**

**例:**
```
give(verb): id=5234
  meaning: "与える"
  courses: [
    { course: "junior", stage: "1" },
    { course: "eiken", stage: "5" },
    { course: "conversation", stage: "a1" },
  ]
```
→ 中学コースで覚えたgiveは、英検・会話コースでも「覚えた単語」として表示される。

**品詞が異なる場合は別エントリ（正当な区別）:**
- address(noun, 住所): id=100
- address(verb, 取り組む): id=101

## 複数品詞の語の扱い

**品詞が複数ある語は、全品詞のmeaningを学習者に表示する。**

- blanket = (名)毛布 (形)全面的な
- handle = (動)扱う・対処する (名)取っ手
- focus = (動)集中する (名)焦点・中心

### データ側の扱い
- 品詞ごとに別エントリ・別IDを持つ
- 各エントリのmeaningは**その品詞の正確な訳**を設定する
- クイズは各エントリ単位で出題される（名詞blanketと形容詞blanketは別の問題）
- **同じ品詞で同じ語は1エントリのみ**（データ統合モデル）

### UI側の扱い
- 単語詳細画面で同一wordの全エントリを集約し、全品詞の意味を表示する
- 主品詞のmeaningを大きく表示、他品詞のmeaningを補足表示
- これにより学習者は語の全体像を把握できる

---

## 語数とティアの考え方

**frequencyTier（頻出度ティア）により、初学者から上級者まで1つのコースで対応する。**

語数の多寡ではなく、**各ティアが目的に応じた網羅性を持つこと**が品質基準。

### ティア別の役割

| ティア | 名前 | 対象ユーザー | 網羅性基準 |
|--------|------|-------------|-----------|
| 1 | 🎯 頻出 | 初学者（確実に合格したい人） | その試験/レベルで必ず出る語を漏れなくカバー |
| 2 | 📚 標準 | 一般学習者（おすすめ） | 高得点・高スコアに必要な語を幅広くカバー |
| 3 | 🔥 発展 | 上級者（満点・完璧を目指す人） | 出題可能性がある語を広くカバー。重箱語もここに含める |

### 固定語数目標は設けない

- vocabulary-policy.mdの設計語数は**参考値**。固定目標として追いかけない
- 「ティア1で合格に必要な語がカバーされているか」「ティア2で高得点を狙えるか」が評価基準
- 語数が多いこと自体は問題ではない（ティアで出し分けるため、初学者にノイズにならない）
- **ティア3として重箱語を含めることで、上級者向けの充実度とダンジョンのゲーム体験を向上させる**

### 上位ステージのティア分布について

英検準1級/1級、TOEIC900、高校3年、会話C1/C2など上位ステージでは、ほとんどの語がティア3になる。
これは正常な挙動である（上位レベルの語は一般英語では低頻出なため）。
上位ステージでティア1が少ない/ゼロでも、それはティア判定の問題ではない。

---

## 禁止事項

- **ティア1/2への重箱語混入禁止**: 試験に出ない専門語・珍しい語をティア1/2に入れてはならない（ティア3には可）
- **カテゴリ不一致禁止**: TOEICコースに学術語、会話コースにビジネス専門語を入れてはならない
- **レベル超過禁止**: 高校2年コースに英検1級語、TOEIC600コースに900点レベル語を入れてはならない
- **meaningの歪曲禁止**: meaning衝突解消のために不正確な訳を入れてはならない
- **安易な英検転用禁止**: 各コースの正しいソースから選定すること
- **同一語の重複エントリ禁止**: 同じ word+partOfSpeech は1エントリのみ。複数コースへの所属はcourses配列で表現する
- **コース別meaning禁止**: 同じ語に対してコースごとに異なるmeaningを設定してはならない（統合データモデルでは構造的に不可能）

## データ品質チェック体制

品質会議で以下の観点からレビューする:

| 観点 | 担当 | チェック内容 |
|------|------|-------------|
| meaningの正確さ | 全員 | 辞書的に正しいか、学習者が間違えて覚えないか |
| 出題品質 | ヘルパーT | クイズとして適切か、選択肢が混乱しないか |
| カリキュラム | マクロファージ | レベル配置は試験基準に合っているか、正しいソースか |
| 記憶定着 | メモリーB | 覚えやすい例文・meaningか |
| 学習者視点 | 一般細胞 | 初学者が見て分かりやすいか |
| データ分析 | 肥満細胞 | 語数・分布・衝突の数値チェック |
| バランス | B細胞 | ゲーム（ダンジョン）で出題バランスが取れるか |
